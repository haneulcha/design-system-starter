# Status Hue 구성 원칙

## 핵심 원리: 색상환 4분할 + 지각/문화적 보정

Status color의 4가지 hue(red, amber, green, blue)는 oklch 색상환의 4분할(quadrant)에 가깝게 배치된다. 이는 4색이 서로 충분히 떨어져 있어야 구분 가능하고 혼동이 없기 때문이다.

```
           90° (yellow/amber)
            │
            │   amber: 65-90°
            │
  180° ─────┼───── 0° (red)
  (cyan)    │         red: 0-30°
            │
  green:    │
  145-163°  │
            │
           270° (blue)
            blue: 254-262°
```

## 실제 시스템 데이터

| System | Red | Green | Amber | Blue | 특징 |
|---|---|---|---|---|---|
| IBM | 25.9° | 148.1° | 90.4° | 262.0° | 거의 90° 간격으로 분포 |
| Spotify | 15.8° | 148.7° | 67.4° | 254.2° | 비슷한 4분할 |
| Stripe | 10.8° | 148.6° | 68.4° | — | red가 pink 쪽으로 shift |
| Resend | 20.6° | — | 84.1° | 258.4° | |
| Notion | — | 145.4° | — | 254.3° | |

### 각 hue의 유효 범위

| Status | 범위 | 중심 | 폭 |
|---|---|---|---|
| Red | 11° ~ 34° | ~21° | 23° |
| Amber | 67° ~ 90° | ~76° | 23° |
| Green | 145° ~ 163° | ~149° | 18° |
| Blue | 254° ~ 262° | ~258° | 8° |

Blue의 범위가 가장 좁다(8°) — "파란색"에 대한 인식 합의가 가장 강하다.

## 완벽한 90° 등분할이 아닌 이유

1. **지각적 균형**: oklch에서 "시각적으로 같은 강도"를 가지는 hue가 균등 분포가 아님. 색상환의 각 영역이 인간의 눈에 동일한 크기로 인식되지 않는다.

2. **문화적 컨벤션**: "위험=빨강, 성공=초록"은 신호등에서 온 것. hue가 이 기대에 맞아야 한다. 만약 수학적으로 완벽한 90° 분할(0, 90, 180, 270)을 쓰면:
   - 0° = red (OK)
   - 90° = yellow (가독성 문제)
   - 180° = cyan (success로 어색)
   - 270° = blue (OK)
   
   문화적 기대와 어긋나는 위치가 생긴다.

3. **Amber > Yellow**: 순수 yellow(~100°)는 밝은 배경에서 가독성이 나쁨. oklch에서 yellow 영역은 높은 lightness에서만 chroma가 유지되므로 warning 표시에 부적합. orange 방향(65-85°)으로 내려야 어두운 텍스트 대비가 확보된다.

4. **Green은 150° 근처**: 순수 green(120°)보다 teal 방향으로 이동한 145-163°가 선호됨. 120°의 pure green은 "cheap" 느낌을 주고, 150° 근처가 더 세련됨.

## 세트로서의 구조

개별 hue의 정확한 각도보다 **4색이 하나의 세트로서 충분히 분리되어 있는가**가 중요하다.

인접 hue 간 최소 간격:

| 인접 쌍 | IBM | Spotify | 최소 간격 |
|---|---|---|---|
| Red → Amber | 64.5° | 51.6° | ~50° |
| Amber → Green | 57.7° | 81.3° | ~58° |
| Green → Blue | 113.9° | 105.5° | ~106° |
| Blue → Red (wrap) | 123.9° | 121.6° | ~122° |

Green ↔ Blue, Blue ↔ Red 간격이 크고, Red ↔ Amber 간격이 가장 작다. 이는 red/amber가 같은 "따뜻한 계열"에 속하지만 여전히 50° 이상 떨어져 있어 구분이 가능하다는 뜻.

## Primary Color와의 관계

Status hue는 primary color에서 파생되지 않는다. 디자인 차원의 고정된 선택이다.

단, **브랜드 hue가 status hue 범위와 겹칠 때는 같은 색을 공유**한다:

| 브랜드 | 겹치는 Status | 처리 |
|---|---|---|
| IBM Blue (262°) | Info Blue (262°) | 같은 색 사용 |
| Spotify Green (149°) | Success Green (149°) | 같은 색 사용 |
| Supabase Green (159°) | Success Green (159°) | 같은 색 사용 |
| Airbnb Red (17°) | Error Red (34°) | **의도적으로 분리** — 브랜드=위험이면 UX 혼동 |

Airbnb 케이스가 중요하다: 브랜드 red(17°)와 error red(34°)를 17° 분리했다. 같은 hue 영역이지만 "이 빨간색은 브랜드, 저 빨간색은 에러"를 구분해야 하므로.

## 현재 스타터의 값

```ts
blue: 258, red: 21, amber: 76, green: 149
```

10개 프로덕션 시스템의 중앙값. 각 유효 범위의 중심에 위치.

## 추후 개선 방향

- Status hue를 사용자가 미세 조정할 수 있게 하되, 유효 범위(±15°)와 인접 hue 간 최소 간격(50°) 제약 조건을 부여
- Primary color hue가 status 범위와 겹칠 때 자동으로 분리하는 로직 (Airbnb 패턴)
- 색맹 접근성 검증: red/green 구분이 어려운 사용자를 위한 대안 hue 세트
